In lab0, we first introduced the Unity game engine and asked you to begin exploring it. In this lab, we’re going to begin introducing some of the fundamental ideas that make Unity unique. This includes one its’ most prominent feature; full C# scripting. Unlike Unreal engine, where you use visual scripting to create your games, Unity requires the creator to create custom scripts. This has the advantage of allowing the creator to tweak the engine as much as possible without having to dive into swaths of complex C++ code.

This approach to scripting comes with unparalleled flexibility. Don’t like Unity’s input manager? Don't use it! Need a custom physics solution? Add it to your game! With Unity, these things are much easier to achieve and with less headaches.

The Inspector

Removing a Component from the Inspector

The Inspector is a great way of easily examining all the components attached to a GameObject. These components range anywhere from Transforms (GameObject position, scale, and rotation) to custom scripts. These components are the basis of the Inspector’s power.

Components can be added and removed at whim. In order to add a component, simply hit the add component button at the very bottom of the Inspector. To remove a component, simply click the down arrow on the component and click delete. However, some components can’t be deleted and Unity will let you know of them.

The Inspector also allows you to change a GameObject’s name, tag, and layer. Tags are solely used for describing a group of objects. A building GameObject can share the tag “Building” with several other GameObjects. This allows for easy categorizing. Layers, on the other hand, are a bit more complicated. Layers are primarily used to ignore GameObjects. Perhaps you have some lighting effect that you don’t want to affect background objects, simply ignore any objects that belong to that layer. Layers can also be used in ignoring objections in a collision detection.

The Hierarchy tab is fairly straight forward when it comes to things, but there’s something that needs to be pointed out.That is, the Hierarchy implements a family-based object structure. Objects can be children of other objects. This means that certain properties are inherited by the children. Namely, transform values and optionally layers. In the above image, we have a basic scene with two GameObjects that are children to another GameObject. If you were to move or scale or rotate the Counter object, both the Storage object and the Toilet object would also be affected. This is useful when needing to manipulate multiple objects at once.

Introduction to C# Scripting

As stated before, Unity uses the C# language in order to create games. C# is very similar to Java syntax wise. A few notable exceptions are noted.

  • A Boolean is simply bool in C#.
  • String becomes string (lowercase)
  • A class does not have to declare that it can throw an exception
  • Packages are imported via the using command (EX: using System.IO)
  • Mutators and Accessors methods can be defined in the field itself

Like Java, C# utilizes a specific runtime in order to run: the .NET Runtime. Unfortunately, Unity implements an older version of this runtime.

Monobehaviours

The vast majority of the time, you’ll be implementing a Monobehaviour. A Monobehaviour is the base class for all Unity scripts. The most basic script contains two methods Start() and Update().


    using UnityEngine;
    using System.Collections;
    using System;

    public class Introduction : Monobehaviour {
        void Start() {
            //Used for initializing the script, sort of like what a constructor does for most objects. 
            //A Monobehaviour never has a constructor.                
        }
        void Update() {
            //where a lot of the magic happens. 
            //Called every frame and is used for everything from grabbing Input to modifying the UI. 
            //If implemented poorly, can easily hamper performance of your game.
        }
    }

Monobehaviours can implement tons of other functions but we’re going to focus on these two for now.

Retrieving Input (Manual)

A game isn't worth anything if you can't interact with this. Fortunately, Unity provides a decent Input Manager for achieving this. For most basic things, and ever we cover in this course, the default Input Manager will work just fine.

There are three basic event types that we're going to worry about; Key, Button, Mouse. Each input event (excluding mouse) has three states; Down, Up, and Held. These events are retrieved via GetKey(), GetKeyDown(), and GetKeyUp() for Keys. The functions are the same naming convention for buttons and mousebuttons.


//Available in Scripts > Lab1
class InputExample : MonoBehaviour {
    KeyCode pushedKey = KeyCode.A;      //The Key we'll be looking for a press
    //Use for initialization
    void Start() { }
    //Main event loop
    void Update() {
        if (Input.GetKeyDown(pushedKey)) {
            Debug.Log("Pushed key!");
        }
    }
}

Examples

So now that we have a general idea of how to write some scripts, let's try some examples.

Popping a balloon

In order to get started, put the Balloon model from the Models folder into your scene. The Balloon has no material, so go ahead and drop a material on it too. Now we need to add our script. Right-click in the Assets tab and click Create > C# Script. Name your script BalloonPop and then drag it onto your Balloon GameObject.

Now double-click on the Script asset and open it in Visual Studio. The created script should contain the Start() and Update() functions that was mentioned early. We're going to use those to implement our code.

The goal: pop the balloon after a certain amount of time.

In order to do this, we have to implement a timer. In order to do this we need a reference to the current time, a start time, and a waitTime. Forunately, Unity already has a way to get the current time through Time.time. So create a private float variable called startTime and a public float variable called waitTime. We want waitTime to public so you can easily change its' value from the Inspector. Within the Start() function, set the startTime value equal to the current Time retrieved by Time.time.

Now we need to check if the allotted time has passed, and we do that with the Update() function. This can really be thought of as taking the change of the startTime to the current time and seeing if it's greater than our waitTime. Effectively a conditional. Writing this would look something like if (currentTime - startTime > waitTime). Throwing it all together should yield the bottom result.


public float waitTime;
private float startTime;
// Use this for initialization
void Start () {
    this.startTime = Time.time;
}
	
// Update is called once per frame
void Update () {
    if (Time.time - startTime >= waitTime) {
        //pop the balloon
    }
}

So how do we pop the balloon? Recall earlier that each GameObject can be Destroyed via the Destroy() method. This is effectively the same thing as a pop. Create a new void function pop() and call it from our timer conditional. Within pop(), we're going to make a single call to GameObject.Destroy(). But Destroy() is a static method and has no knowledge of our current object, so we have to give it a reference to the GameObject we want to Destroy. If we pass this as an argument, we only destroy the script calling the Destroy function. But if we pass this.gameObject, we get the reference to the parent GameObject we need to destroy. So our pop() function should look like this.


void pop() {
    GameObject.Destroy(this.gameObject); 
}

Challenge: Wait longer to pop the balloon.

How could we change our code so that we wait a longer interval to pop the balloon, maybe say 30 seconds?

Currently, our script is pretty boring. We wait X seconds, and then destroy the Object. What if we want to add some motion to our Balloon? Maybe our Balloon flies for a couple of seconds and then pops? This can be achieved along the same lines by just adding to our Update() function. Recall from above that Transform.position gives us the current position of the object. So what if we incrementally added onto the position of our balloon every frame?In the Update() function, add a new Vector3 to our current position and store it as the current position. This should look something like this.


// Update is called once per frame
void Update () {
    if (Time.time - startTime >= waitTime) {
        this.transform.position = this.transform.position + new Vector3(0,0,1);
        pop();
    }
}

Script Communication

Great, so we can make basic scripts and populate our scenes with them but what if we need them to talk to each other? Unity actually gives us a couple of ways of doing this. We can do this statically (known at compile-time) or dynamically. We use static base communication wheever we know beforehand what object we'll need to call, such as updating the main camera or adding points to a scoreboard. We do this dynamically whenever we don't know the exact object we'll be calling, such as hitting a target.

The way to do this is by having a reference to the object you're calling, and getting a component from that object using GameObject.GetComponent(Type). Once, you get the Object reference you can call any public method in that component. You can use GameObject.SendMethod as well, but it can be expensive to run.

More Examples

Revisiting our balloon example, what if we wanted to make the balloon pop after some arbitrary input. We could have a InputManager GameObject that handles all of our input and then calls objects to do our biding. So let's do just that. Create a new Empty GameObject called InputManager and add it to our scene. Next create a new script for the GameObject called Manager. The Manager script will handle all of our input, and call external methods. Open the Manager Script for editing. Now we need a way to grab the reference to the Balloon. Remember from above that public variables are shown in the Inspector, and that a GameObject field has a GameObject picker associated with it. So create a public GameObject variable named BalloonRef, and use the picker to grab our Balloon object.

If you recall, our BalloonPop script has no public methods, we need to change that. Simply add public to void pop() and now the method is public. In our script, we have a reference to the GameObject, but we don't have a reference to the BalloonPop Script Component. We need to use GameObject.GetComponent() and then store the reference for later use. This can be done as follows.


public GameObject BalloonRef;
private BalloonPop popScript;
// Use this for initialization
void Start () {
    popScript = BalloonRef.GetComponent("BalloonPop") as BalloonPop;
    //popScript = BalloonRef.GetComponent<BalloonPop>();
    //Same method as above, just faster
}

Just like that, we've taken care of all the hard bits. Now we just have to modify our Input Script from before, and call the pop() function. This can be done like so.


private KeyCode popKey = KeyCode.P;
// Update is called once per frame
void Update () {
	if (Input.GetKeyDown(popKey)) {
    BalloonPop.pop();
    }
}

Challenge: Change the wait time remotely

We have a reference to the BalloonPop script, so what would we have to do change the waitTime variable in it from another script?

There's a bug with the above script, can you guess what it is? Specifically, there's a problem with the initilization in the Start() function. What happens if we forget to set the GameObject reference in the Inspector or it somehow becomes unassigned? The script doesn't check for that and quickly fails. In order to prevent this, public variables assigned in the Inspector should always be checked before using.


public GameObject BalloonRef;
private BalloonPop popScript;
// Use this for initialization
void Start () {
    if (BalloonRef == null) {
        Debug.LogError("Balloon object hasn't been set!");
    }
}

You should also check the variable before using it in the Update() function as well.

As mentioned before, there is another way of getting an object reference. You can use GameObject.Find() to get any GameObject within the Scene. Why don't we adapt our Balloon example to use this method? The only thing we have to change is in our Manager script. First, we need to make BalloonRef private so the Inspector doesn't attempt to assign values to it. Next we need to assign the variable with our retrieved object. This can be easily done with the following.


private GameObject BalloonRef;
private BalloonPop popScript;
private KeyCode popKey = KeyCode.P;
// Use this for initialization
void Start () {
    BalloonRef = GameObject.Find("Balloon");
    if (BalloonRef == null) {
        Debug.LogError("Balloon object hasn't been set!");
    }
    else {
        popScript = BalloonRef.GetComponent("BalloonPop") as BalloonPop;
    }
    //popScript = BalloonRef.GetComponent();
    //Same method as above, just faster
}

Notice how we still check the returned value for null? This happens because Find() doesn't indicate any errors. It simply returns null if it can't find the GameObject. There's also another problem with Find; it's expensive. The Find() function works by searching the entire scene until it finds the name of the GameObject you requested. This is okay if you only ever do it once and you don't have a large scene. If you don't follow this advice, you could see some serious FPS issues with your game.
